# 全局开发协作规则

## 总体目标

- 默认自主完成代码阅读、实现、测试、修复和结果总结。
- 优先完成当前任务所需的最小改动，不顺手重构无关代码。
- 变量名、函数名、类名使用英文；代码注释、方案说明和总结使用中文。
- 注释只用于复杂逻辑、关键业务规则、兼容处理和易踩坑点，重点解释“为什么”。

## 任务分级

### 小任务和简单报错

满足大部分以下条件时可直接处理：修改范围清楚、涉及文件较少、无架构调整、无敏感数据或高风险操作。

- 直接阅读相关代码并实现。
- 主动补充或更新必要测试。
- 自主运行相关测试、Lint、类型检查或构建。
- 不启动子智能体。
- 完成后给出简短总结：修改文件、改动内容、验证结果、遗留问题。

### 中大型任务

出现以下任一情况时按中大型任务处理：需求存在多种方案、跨多个模块、涉及架构或数据模型、预计修改 3 个以上核心文件、需要新增完整功能、影响范围不容易回退。

1. 先阅读相关代码和配置。
2. 给出合理方案，包括目标、实现步骤、影响范围、关键取舍、风险和验证计划。
3. 与用户讨论并确认方案后再开始实现。
4. 方案确认后立即创建开发复盘草稿，并在实现完成、测试结束时继续更新。
5. 自主编码、测试和修复。
6. 完成后执行代码审查；敏感功能同时执行安全检查。
7. 输出博客式复盘，并保存到项目文档中。

## 需求不完整时

- 不直接根据模糊描述大规模编码。
- 先结合现有代码给出一套推荐方案，明确假设和待确认点。
- 与用户讨论完善计划并得到确认后再实现。
- 小修改只有轻微歧义时，可以采用最小、可回退的合理假设，并在总结中说明。

## 自主操作范围

可以自主执行：

- 阅读、创建、修改和删除当前项目内的普通代码、测试和文档文件。
- 运行项目、测试、Lint、类型检查和构建命令。
- 查看 `git status`、`git diff`、`git log` 等只读信息。
- 根据新的错误信息调整实现并继续修复。

以下操作必须先向用户确认，并说明操作内容、目的、命令、影响范围、预期结果、风险和回退方式：

- 安装、删除或升级依赖。
- 修改依赖锁文件所对应的依赖集合。
- 修改或删除 `.env`、密钥、凭据和环境变量。
- 修改或删除项目目录外的文件、系统目录或全局配置。
- 数据库迁移、建表、删表、修改数据结构或批量修改数据。
- 创建或删除 Git 分支、`git commit`、`git pull`、`git push`。
- Git 回滚、`reset`、`rebase`、强制切换、强制推送等可能丢失改动的操作。
- 其他不可逆或明显扩大影响范围的操作。

## 子智能体与 Agent Team

- 默认不使用子智能体。
- 只有复杂任务确实需要大量代码探索、独立审查或可并行分析时才考虑使用。
- 最多同时启用 3 个。
- 启用前必须告诉用户：准备启用几个、各自负责什么、为什么需要、预计增加哪些调用；获得确认后才能启动。
- 使用完成后停止全部子智能体，并清理临时 Agent Team 状态。
- 最终总结中说明是否使用了子智能体及其产出。

## 测试与修复

- 新功能默认主动编写或补充测试。
- 优先运行与本次修改直接相关的测试，再根据影响范围决定是否运行完整测试套件。
- 失败后根据错误信息调整方案，避免无变化地重复同一命令。
- 自动重试次数由问题复杂度决定，但最多 5 轮。
- 5 轮内仍未解决时停止，说明根因判断、已尝试方法、每次结果、当前阻塞点、可选方案和推荐下一步。
- 不通过删除测试、关闭检查或吞掉异常来伪造通过。

## 审查与安全

以下情况自动执行代码审查：

- 中大型任务完成后。
- 涉及登录、权限、支付、数据库、文件上传、外部输入或敏感数据。
- 用户准备执行 Git 提交前。

安全检查重点关注：硬编码密钥、注入、越权、认证绕过、敏感信息泄漏、不安全文件路径、SSRF、XSS、错误的事务或并发处理。

只报告有明确代码位置、具体触发方式和可解释后果的问题，避免制造低价值告警。

## 项目导读

- 第一次处理陌生项目时，读取 `.claude/ecc-init-state.json`。
- 如果 `code_tour_completed` 为 `false`，自动执行一次代码导读：识别技术栈、目录结构、启动入口、主要调用链、配置、数据存储和外部依赖。
- 将结果写入 `docs/PROJECT_OVERVIEW.md`，完成后把状态改为 `true`。
- 已完成且项目结构没有明显变化时不重复导读。

## 开发复盘

- 中大型任务使用 `docs/DEVELOPMENT_LOG.md` 作为索引，并在 `docs/dev-notes/` 中为每个任务创建单独文章。
- 文章在三个阶段更新：方案确认后、实现完成后、测试结束后。
- 文章主要写给未来回看的自己：记录为什么做、为什么选择该方案、怎么实现、遇到什么困难、如何解决、测试结果和学到什么。
- 开头简要列出技术栈、关键依赖版本、开发环境、启动和测试命令。
- 小修改不创建长文，只输出简短总结。

## 完成标准

在声称任务完成前，确认：

- 需求已实现，且没有扩大到无关范围。
- 相关测试已补充并运行。
- 已说明实际执行的验证和未验证内容。
- 中大型或敏感任务已完成必要审查。
- 删除的项目文件已在总结中说明原因。
- 中大型任务的开发日志已更新。
