# 验证闭环

## 原则

- 先运行最相关、成本最低的验证，再扩大范围。
- 只运行项目中真实存在的命令。
- 不因测试失败而删除测试、关闭规则或吞掉异常。
- 安装缺失依赖前必须向用户解释并确认。

## 验证顺序

1. **相关测试**：优先运行本次修改涉及的测试文件或模块。
2. **类型检查**：项目已配置时执行。
3. **Lint/格式检查**：项目已配置时执行；自动格式化前先确认不会扩大无关 diff。
4. **集成或端到端测试**：只有影响跨模块流程时执行。
5. **构建或启动验证**：确认应用能构建或启动到预期阶段。
6. **变更检查**：执行 `git status` 和 `git diff --stat`，确认没有无关改动。

## 失败处理

- 读取完整错误和相关代码，先判断根因。
- 每一轮必须基于新证据调整方案，不重复无变化的命令。
- 由复杂度决定重试次数，最多 5 轮。
- 达到上限后停止，并报告：
  - 根因判断；
  - 已尝试的方法；
  - 每轮结果；
  - 当前阻塞；
  - 可选方案及推荐下一步。

## 输出格式

```markdown
## 验证结果
- 相关测试：PASS / FAIL / 未配置
- 类型检查：PASS / FAIL / 未配置
- Lint：PASS / FAIL / 未配置
- 构建或启动：PASS / FAIL / 未执行
- 变更范围：X 个文件

## 未验证内容

## 遗留问题
```
