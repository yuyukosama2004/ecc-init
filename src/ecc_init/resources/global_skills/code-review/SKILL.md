# 代码审查

## 何时使用

- 中大型任务实现并验证后。
- 涉及登录、权限、支付、数据库、文件上传或外部输入。
- 用户准备提交 Git 前。

普通小修改不额外触发，除非风险明显。

## 审查流程

1. 查看 `git status`、`git diff --stat` 和 `git diff`。
2. 阅读变更文件的完整上下文、调用方和相关测试。
3. 检查正确性、边界条件、错误处理、并发/事务、性能和可维护性。
4. 检查是否存在无关修改、死代码、调试输出或遗漏测试。
5. 只报告置信度高于 80% 的真实问题。

## 报告门槛

每个问题必须同时具备：

- 精确文件和代码位置；
- 具体触发输入或状态；
- 可解释的错误结果；
- 现有校验、类型或框架默认行为为何不能阻止它。

缺少上述证据时，降低严重度或不报告。允许返回零问题。

## 严重度

- **CRITICAL**：可能导致数据丢失、权限绕过、密钥泄漏或远程执行。
- **HIGH**：稳定复现的业务错误、安全漏洞或重要数据不一致。
- **MEDIUM**：特定条件下的错误、明显维护风险或性能问题。
- **LOW**：有价值但不阻塞的改进；避免纯风格偏好。

## 输出格式

```markdown
## 审查结论
APPROVE / NEEDS_CHANGES

## 问题
### [严重度] 标题
- 位置：`path:line`
- 触发条件：
- 后果：
- 建议修复：

## 测试缺口

## 可选改进
```
