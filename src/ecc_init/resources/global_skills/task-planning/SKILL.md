# 中大型任务规划

## 触发条件

出现以下任一情况时使用：

- 需求不完整，需要先结合现有项目补全。
- 涉及多个模块或多个核心文件。
- 需要调整架构、数据模型、接口契约或状态流转。
- 存在多个可行方案，取舍会影响维护成本。
- 失败可能造成数据、权限或兼容性问题。

小修改、明确报错、单文件局部调整不使用本 Skill。

## 工作流程

1. 阅读 README、入口、相关实现、测试和配置，不根据文件名猜测。
2. 用一句话重述目标，并列出关键约束。
3. 最多给出 2—3 个真正可行的候选方案。
4. 推荐其中一个，说明为什么适合当前项目。
5. 给出不超过 7 步的实施计划。
6. 说明预计修改文件、数据影响、测试计划和回退方式。
7. 列出仍需用户确认的事项。
8. 等待确认，不提前大规模编码。

## 输出格式

```markdown
## 目标与现状

## 推荐方案

## 为什么选择它

## 实施步骤
1. ...

## 预计影响
- 文件：
- 接口或数据：
- 兼容性：

## 验证计划

## 风险与回退

## 需要确认
```

## 约束

- 不为了展示能力而引入新框架或复杂抽象。
- 安装依赖、数据库迁移、修改环境变量和 Git 写操作必须单独向用户确认。
- 方案确认后触发 `dev-retrospective` 创建任务草稿。
